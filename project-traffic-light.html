A smart traffic light controller that gives the highway right-of-way by default and only flips to the farm-road when a vehicle is detected by an ultrasonic sensor.

The problem

Real traffic lights at small intersections are dumb — they cycle on a fixed schedule whether anyone's at the light or not. That wastes everyone's time on the busy road. The standard fix is "actuated control": detect cars on the side road and only give them green when needed.

I implemented this as a college Computer Engineering project, starting from a Logisim digital-logic version and porting the same FSM to ESP32 firmware.

The finite state machine

Four states control the lights. The transitions handle the interesting edge cases: minimum hold time on farm-road green (so cars don't pass too quickly), maximum hold time (so the highway always gets a turn), and early exit when the car leaves.

StateHighwayFarm-roadTransition
HGGreenRedStay until car detected
HYYellowRedAfter 2 seconds
FGRedGreenMin 3 s, leave when car gone, max 16 s
FYRedYellowAfter 2 seconds

Hardware

  • ESP32-S3-N16R8 dev board
  • HC-SR04 ultrasonic sensor (5V supply, ECHO via 1k/2k voltage divider for safe 3.3V GPIO)
  • 6 LEDs (red/yellow/green for each road)
  • 6× 220 Ω current-limiting resistors
  • 3× 1 kΩ resistors for the divider (two in series for the 2 kΩ leg)

Why a voltage divider?

The HC-SR04 outputs 5 V on its ECHO pin, but the ESP32-S3's GPIO is 3.3 V tolerant. Plugging 5 V directly into a 3.3 V pin damages the chip over time. A voltage divider with a 1 kΩ / 2 kΩ ratio drops the signal to a safe ~3.3 V: 

V_out = V_in × R2 / (R1 + R2)
      = 5 V × 2kΩ / (1kΩ + 2kΩ)
      ≈ 3.33 V

Software design

The firmware uses non-blocking timing with millis() instead of delay(), so the FSM transitions react instantly to sensor changes while still holding states for fixed durations. The HC-SR04 is polled every 200 ms with a 30 ms pulseIn() timeout — long enough for ~5 m of range, short enough that a missing echo doesn't stall the loop. 

void loop() {
  if (millis() - lastSenseAt >= SENSE_PERIOD) {
    lastSenseAt = millis();
    float d = readDistanceCM();
    carPresent = (d > 0 && d < CAR_DISTANCE_CM);
  }

  unsigned long elapsed = millis() - stateStart;

  switch (state) {
    case HG:
      if (carPresent) enterState(HY);
      break;
    case HY:
      if (elapsed >= YELLOW_TIME) enterState(FG);
      break;
    case FG:
      if (elapsed >= FG_MAX_TIME) enterState(FY);
      else if (elapsed >= FG_MIN_TIME && !carPresent) enterState(FY);
      break;
    case FY:
      if (elapsed >= YELLOW_TIME) enterState(HG);
      break;
  }
}

What I learned

  • Hardware/software equivalence. The same FSM works as a Logisim circuit (D flip-flops + counter) and as embedded C++. Different implementations, identical behavior — and seeing it both ways made the FSM concept click in a way the textbook didn't.
  • Voltage levels matter. "It just works" until you plug 5 V into a 3.3 V GPIO and silently degrade the chip. Always check the datasheet.
  • Non-blocking timing. delay() is a beginner trap. The millis() pattern lets multiple things happen at once on a single-threaded MCU.
  • Debug with what you have. When Serial Monitor wasn't working, I used the LEDs themselves as a debug output to confirm the sensor was reading correctly.
