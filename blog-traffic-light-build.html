Building an ESP32 traffic
light from scratch.

POSTEDMay 2026
READ~10 min
TAGSESP32 · Embedded · Build Log

A complete build log of my ESP32-S3 ultrasonic traffic light — including every dead end, debugging session, and "why isn't this working" moment that didn't make it into the project page.

The starting point

In my Computer Engineering coursework I'd already designed a traffic-light finite state machine in Logisim Evolution: D flip-flops for the state register, a counter for timing, ROM for the output decoder. It worked in simulation. The natural next step was building the same FSM in real hardware with an ESP32-S3 and an ultrasonic sensor.

Spoiler: the FSM was the easy part.

Step 1 — Get the sensor working alone

Before wiring up six LEDs on top of a sensor I didn't understand, I built a standalone HC-SR04 test sketch. Just trigger the pulse, read the echo, print the distance. About 25 lines of code. This turned out to be the best decision I made in the project.

First try? Pulse duration of zero, every reading. The sensor was wired, powered, and silent. The next two hours were a tour of every common HC-SR04 mistake.

Mistake 1: Wrong VCC

The HC-SR04 needs 5 V. The ESP32-S3 has both a 5 V and a 3.3 V pin. Mine was on 3.3 V. Moved it to 5 V — still no echoes.

Mistake 2: The voltage divider

The sensor's ECHO pin outputs 5 V, but the ESP32 GPIO can only handle 3.3 V. A standard fix is a 1 kΩ / 2 kΩ voltage divider that drops the 5 V down to ~3.3 V. I had it wired correctly... or so I thought.

Turned out the resistors were both legs in the same row of the breadboard, so they weren't actually doing anything. The signal was flowing straight around them. Once I fixed the divider so each resistor bridged two different rows — readings appeared.

Mistake 3: Only 1 kΩ resistors available

My kit had 20× 1 kΩ resistors but no 2 kΩ. The fix: two 1 kΩ in series equals 2 kΩ. Three resistors instead of two, identical electrical behavior:

Step 2 — Add the LEDs

With the sensor reporting accurate distances, I moved on to the LEDs. Six of them — red, yellow, green for highway and farm-road. Each with a 220 Ω current-limiting resistor.

First upload of the full traffic light code: only the red LED on highway lit up. The other five — nothing. Sensor was working, code was running, but five LEDs refused to turn on. I rewired everything. Three times. Same result.

What actually broke it

The issue was the GND rail. I'd run the sensor's ground to one rail and forgotten that some breadboards have a split in the middle of the power rails — the top half and bottom half aren't connected unless you bridge them with a wire.

The LEDs were grounding to the bottom rail. The ESP32's GND wire was on the top rail. They looked connected, but electrically they weren't. Adding a single jumper from the top rail to the bottom rail fixed five LEDs simultaneously.

Step 3 — Serial Monitor not displaying

With everything else working, I noticed Serial Monitor stopped printing. The LEDs were behaving correctly, but the debug output had gone silent.

The fix is an ESP32-S3-specific quirk. There's a setting called USB CDC On Boot in Arduino IDE's Tools menu that controls whether the chip prints over the USB serial port at boot. The default is sometimes "Disabled," which silently drops every Serial.print(). Switching it to "Enabled" and re-uploading made the messages appear.

I would never have guessed this from the symptoms alone. The error message was: nothing.

Step 4 — The actual FSM was easy

Once the hardware worked, the four-state FSM came together in about 30 minutes. Non-blocking timing with millis() instead of delay(), a switch on the state enum, and a helper function to centralize state transitions.

The most subtle bit was the farm-road green state, which has three exit conditions: minimum hold of 3 seconds (so cars don't pass too fast), maximum hold of 16 seconds (so the highway gets a turn), and early exit when the car leaves after the minimum is met. 

case FG:
  if (elapsed >= FG_MAX_TIME) {
    enterState(FY);
  } else if (elapsed >= FG_MIN_TIME && !carPresent) {
    enterState(FY);
  }
  break;

What I'd do differently

  • Test the breadboard's power rails first. Before placing any components, jumper the top rail to the bottom rail and make sure both sides connect to ESP32 GND.
  • Build incrementally. Sensor first, then one LED, then six. The "all at once" approach hides which step broke things.
  • Configure USB CDC before uploading. If you're on ESP32-S3, set USB CDC On Boot = Enabled before your first upload.

Takeaways

The thing about embedded projects is that "wrong" doesn't mean "broken." Every mistake I hit was something the textbook had probably mentioned in passing — voltage divider topology, breadboard power rail layout, USB CDC setting. None of it would have been a surprise if I'd been paying closer attention to the manual.

But until you wire it up wrong yourself, those details don't stick. That's why projects matter more than reading.
